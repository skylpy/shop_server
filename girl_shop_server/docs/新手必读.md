# Girl Shop 新手必读

这份文档专门写给第一次接手本项目的同学。

目标只有两个：

- 让你快速看懂每个模块是干什么的
- 让你看懂一个接口从请求进入到返回结果的完整流程

如果你刚接触 `Node.js + Express + MongoDB` 项目，建议先读这份文档，再去看业务代码。

## 1. 先记住这个项目的主结构

你可以把当前项目先理解成一句话：

- `app.js` 负责把所有零件装起来
- `routes` 负责定义“哪个 URL 交给谁处理”
- `controllers` 负责接收请求并返回响应
- `services` 负责真正的业务逻辑
- `models` 负责和 MongoDB 集合对应
- `middlewares` 负责鉴权、错误处理等公共流程
- `utils` 负责放通用小工具
- `config` 负责配置与初始化

## 2. 每个模块的作用说明

### 2.1 app.js

作用：

- 项目启动入口
- 加载 `.env` 和 `.env.local`
- 初始化 Express
- 注册 JSON 解析、中间件、静态资源、路由、错误处理
- 启动前调用数据库初始化逻辑

新手理解方式：

- 它本身不写业务细节
- 它更像“总装配台”
- 你要找某个接口具体做了什么，通常不要从这里往下写业务，而是顺着 `routes -> controllers -> services` 去看

### 2.2 config

当前主要文件：

- `config/db.js`

作用：

- 统一管理 MongoDB 连接
- 避免在多个文件里重复写数据库连接代码

新手理解方式：

- 以后如果数据库地址、连接参数要改，优先看这里

### 2.3 models

当前主要文件：

- `models/index.js`

作用：

- 集中注册所有 Mongoose Model
- 让 `settings`、`users`、`products`、`orders`、`operation_logs` 等集合都有统一入口

新手理解方式：

- 它解决的是“代码里怎么拿到 MongoDB 集合”
- service 层需要查数据库时，会从这里拿 model

### 2.4 routes

当前主要文件：

- `routes/index.js`
- `routes/adminRoutes.js`
- `routes/userRoutes.js`
- `routes/shopRoutes.js`

作用：

- 把请求路径和控制器方法对应起来
- 例如 `/api/admin/login` 交给后台控制器处理
- 例如 `/api/user/profile` 先走用户鉴权中间件，再进入用户控制器

新手理解方式：

- 看到一个接口路径，先到 `routes` 里找
- `routes` 只负责分发，不应该塞大段业务逻辑

### 2.5 controllers

当前主要文件：

- `controllers/adminController.js`
- `controllers/userController.js`
- `controllers/shopController.js`
- `controllers/systemController.js`

作用：

- 接收请求参数
- 调用对应 service
- 用统一格式返回 `code/message/data`

新手理解方式：

- controller 是“接口门面层”
- 它知道“调用哪个 service”
- 但它通常不负责复杂业务判断

举例：

- `adminController.login` 会读取 `req.body`
- 调用 `adminService.login`
- 最后通过 `success(res, data)` 返回给前端

### 2.6 services

当前主要文件：

- `services/adminService.js`
- `services/userService.js`
- `services/shopService.js`
- `services/databaseService.js`
- `services/securityService.js`
- `services/seedDataService.js`
- `services/operationLogService.js`

作用：

- 放核心业务逻辑
- 放数据读写逻辑
- 放 JWT、安全校验、种子数据等公共业务能力

新手理解方式：

- 真正“做事”的地方主要就在 service
- 如果你要改登录逻辑、改商品列表逻辑、改注册逻辑，通常都是改这里

各文件可这样理解：

- `adminService.js`
  作用：后台管理业务，包含管理员登录、商品管理、用户管理、分类管理、订单管理、轮播管理
- `userService.js`
  作用：商城用户注册、登录、个人资料查询
- `shopService.js`
  作用：兼容旧 Flutter 商城接口，返回首页、分类、热卖、商品详情等数据
- `databaseService.js`
  作用：封装数据库读取、初始化、整库写回、重置演示数据
- `securityService.js`
  作用：封装密码哈希、JWT 签发、JWT 校验、请求头 token 提取
- `seedDataService.js`
  作用：提供初始演示数据
- `operationLogService.js`
  作用：统一写入和查询操作日志、审计记录
- `uploadService.js`
  作用：统一处理图片上传目录和上传结果数据

### 2.7 middlewares

当前主要文件：

- `middlewares/auth.js`
- `middlewares/errorHandler.js`
- `middlewares/legacyProxy.js`
- `middlewares/auditTrail.js`
- `middlewares/uploadImage.js`

作用：

- 在请求进入 controller 前后做统一处理

各文件作用：

- `auth.js`
  作用：校验后台管理员 JWT 和商城用户 JWT
- `errorHandler.js`
  作用：统一处理抛出的错误，避免每个接口都自己写一遍 `try/catch`
- `legacyProxy.js`
  作用：兼容旧前端传入的 `proxy` 参数
- `auditTrail.js`
  作用：在请求完成后自动把本次操作写入审计日志
- `uploadImage.js`
  作用：处理图片上传、文件类型校验和文件大小限制

新手理解方式：

- middleware 就像“请求处理流水线上的关卡”
- 先过关，再进入 controller

### 2.8 utils

当前主要文件：

- `utils/response.js`
- `utils/asyncHandler.js`
- `utils/logger.js`
- `utils/dateTime.js`
- `utils/shopMapper.js`
- `utils/appError.js`

作用：

- 放可以被多个模块复用的小工具

各文件作用：

- `response.js`
  作用：统一成功和失败返回格式
- `asyncHandler.js`
  作用：包装异步 controller，自动把异常交给错误处理中间件
- `logger.js`
  作用：统一打印日志
- `dateTime.js`
  作用：生成项目里统一的时间字符串
- `shopMapper.js`
  作用：把数据库里的商品、分类、图片信息转换成旧 Flutter 前端需要的结构
- `appError.js`
  作用：定义业务错误对象，方便 service 里直接抛出明确错误

### 2.10 操作日志和审计记录模块

这是这次新增的重要模块。

它的目标很简单：

- 以后谁做了什么操作，可以查出来
- 操作成功还是失败，可以查出来
- 某次问题发生时，可以根据日志倒推是谁、什么时候、调了哪个接口

现在的实现方式是：

- 请求开始时，controller 会给当前请求挂上 `req.auditLog`
- 请求结束后，`middlewares/auditTrail.js` 会自动把日志写入 `operation_logs`
- 真正的写库和查询逻辑在 `services/operationLogService.js`

当前已经记录的典型事件包括：

- 管理员登录、退出登录
- 商城用户注册、登录、退出登录
- 后台商品、用户、分类、订单、轮播图等关键写操作

前端查看方式：

- 后台登录成功后，左侧导航新增了“操作日志”页面
- 页面本质上调用的就是 `GET /api/admin/operation-logs`
- 每条日志支持点击“详情”查看请求明细和返回明细
- 返回数据落库前会自动对 `token` 等敏感字段做脱敏

### 2.11 图片上传模块

这是后台新增的通用上传能力。

它解决的问题是：

- 管理员不用手工输入图片路径
- 可以直接在后台选择图片上传
- 上传后系统自动把图片路径回填到表单

当前上传保存位置：

- `public/images/uploadImage/`

后端上传接口：

- `POST /api/admin/upload-image`

当前已经接入上传的后台表单：

- 商品封面图
- 商品图库
- 分类图片
- 轮播图片

查看日志的后台接口：

- `GET /api/admin/operation-logs`

### 2.9 lib 和 router

作用：

- 这是兼容层，不是新的主开发目录
- 它们现在主要用于兼容项目里可能残留的旧引用方式

新手理解方式：

- 新增功能优先写到 `routes / controllers / services / utils / middlewares`
- 不建议再继续把新逻辑堆回旧的 `lib` 和 `router`

## 3. 调用接口后的整个流程说明

这一节用最容易理解的方式讲：一个接口从“被调用”到“返回结果”到底经过了什么。

你可以把整个流程记成这条主线：

1. 前端或客户端发起 HTTP 请求
2. `app.js` 接住请求
3. 请求进入 `routes`
4. 如果有鉴权要求，先经过 `middlewares`
5. 然后进入 `controller`
6. `controller` 调用 `service`
7. `service` 处理业务逻辑并读写数据库
8. `controller` 用统一格式把结果返回
9. 如果中途报错，交给 `errorHandler` 统一处理

### 3.1 先理解一次完整请求是怎么走的

假设前端调用：

- `POST /api/admin/login`

它在项目里的执行顺序是：

1. 请求先进入 `app.js`
   这里已经提前注册好了 JSON 解析、中间件和总路由
2. `app.js` 把请求交给 `routes/index.js`
3. `routes/index.js` 发现路径是 `/api/admin/*`
   所以继续交给 `routes/adminRoutes.js`
4. `routes/adminRoutes.js` 发现是 `/login`
   所以调用 `adminController.login`
5. `adminController.login` 读取 `req.body`
   然后调用 `adminService.login`
6. `adminService.login` 开始做真正的业务判断：
   校验用户名密码、读取数据库、生成 JWT、更新最近登录时间
7. `adminService.login` 把结果返回给 controller
8. `adminController.login` 调用 `success(res, data)`
   最终返回统一格式的 JSON 给前端

如果第 6 步校验失败：

- `adminService.login` 会抛出业务错误
- 错误会被 `asyncHandler` 捕获
- 然后交给 `middlewares/errorHandler.js`
- 最终返回统一的错误格式

如果这个请求配置了审计日志：

- `auditTrail` 会在响应结束后把本次操作写入 `operation_logs`
- 成功就记为 `success`
- 失败就记为 `failure`

### 3.2 后台登录接口的完整流程

接口：

- `POST /api/admin/login`

执行过程：

1. 前端把管理员账号密码提交到后端
2. `routes/adminRoutes.js` 把请求交给 `adminController.login`
3. `adminController.login` 调用 `adminService.login`
4. `adminService.login` 读取数据库中的 `adminUsers`
5. 校验账号是否存在、密码哈希是否匹配、状态是否启用
6. 校验通过后用 `securityService.createAuthToken` 生成 JWT
7. 更新管理员的 `lastLoginAt`
8. 返回 `token + expiresAt + profile`
9. 前端把 token 存起来，后续请求放到 `Authorization` 请求头
10. 请求结束后，系统会自动写一条“管理员登录成功”的审计记录

新手记忆重点：

- 登录接口本身不直接查 MongoDB model
- controller 不处理密码逻辑
- 真正的判断都在 service

### 3.3 后台受保护接口的完整流程

以这个接口为例：

- `GET /api/admin/products`

执行过程：

1. 前端请求 `/api/admin/products`
2. 请求先进入 `routes/adminRoutes.js`
3. 因为这个接口在鉴权中间件保护范围内
   所以会先经过 `middlewares/auth.js`
4. `authenticateAdmin` 从请求头里取出 `Authorization: Bearer <token>`
5. 调用 `securityService.verifyAuthToken` 校验 JWT
6. 如果 token 不合法或过期，直接返回错误
7. 如果 token 合法，把解析后的管理员信息挂到 `req.auth`
8. 然后才进入 `adminController.listProducts`
9. controller 再调用 `adminService.listProducts`
10. service 从数据库读取商品、分类信息，做筛选和格式整理
11. controller 把结果返回给前端
12. 如果这是关键写操作，`auditTrail` 会在响应结束后自动记日志

新手记忆重点：

- 鉴权不是 controller 做的
- 鉴权是在 middleware 里提前完成的
- controller 可以直接从 `req.auth` 里拿到已验证的登录信息

### 3.4 商城用户登录接口的完整流程

接口：

- `POST /api/user/login`

执行过程：

1. 客户端提交手机号和密码
2. `routes/userRoutes.js` 把请求交给 `userController.login`
3. `userController.login` 调用 `userService.login`
4. `userService.login` 在 `users` 集合中查找手机号
5. 校验用户状态、密码哈希、是否已注册
6. 通过后生成用户 JWT
7. 更新用户最近登录时间
8. 返回 `token + expiresAt + profile`
9. 系统自动补一条“商城用户登录成功”的审计记录

### 3.5 获取当前商城用户资料的完整流程

接口：

- `GET /api/user/profile`

执行过程：

1. 客户端在请求头带上 `Authorization: Bearer <token>`
2. `routes/userRoutes.js` 先让请求经过 `authenticateUser`
3. `authenticateUser` 校验 token 并把 `userId` 挂到 `req.auth`
4. 然后进入 `userController.profile`
5. `userController.profile` 调用 `userService.getProfile`
6. `userService.getProfile` 去数据库读取当前用户
7. 如果用户不存在或被禁用，抛出错误
8. 如果正常，返回脱敏后的用户资料

### 3.6 旧 Flutter 商城接口的完整流程

以这个接口为例：

- `POST /getHomePageContent`

执行过程：

1. Flutter 前端发请求到旧接口地址
2. `routes/index.js` 把它交给 `routes/shopRoutes.js`
3. `routes/shopRoutes.js` 调用 `shopController.getHomePageContent`
4. `shopController.getHomePageContent` 调用 `shopService.getHomePageContent`
5. `shopService.getHomePageContent` 读取数据库里的轮播、商品、楼层、分类
6. 再通过 `utils/shopMapper.js` 把数据转换成旧前端熟悉的字段结构
7. 返回给 Flutter 前端

新手记忆重点：

- 旧商城接口之所以还能继续用，是因为我们保留了旧路径
- 只是内部实现已经切到新的分层结构里了

### 3.7 数据库在整个流程里的位置

很多新手会问：

- “到底是谁在查数据库？”

答案是：

- 主要是 `services` 在查数据库
- 更准确地说，是 service 调用 `services/databaseService.js`
- `databaseService` 再通过 `models` 和 MongoDB 交互

你可以把数据库层理解成这样：

- `models`
  负责“我有哪些集合可以用”
- `databaseService`
  负责“怎么读、怎么写、怎么初始化”
- `adminService / userService / shopService`
  负责“什么时候读、为什么写、写什么业务数据”

### 3.8 错误处理是怎么统一起来的

以“用户密码错误”为例：

1. `userService.login` 发现密码不对
2. 直接抛出 `new AppError("账号或密码错误", 401)`
3. `asyncHandler` 自动捕获这个异常
4. 异常交给 `middlewares/errorHandler.js`
5. 最终统一返回：

```json
{
  "code": "1",
  "message": "账号或密码错误",
  "data": null
}
```

这样做的好处是：

- controller 代码更干净
- 每个接口的错误格式一致
- 前端更容易统一处理
- 审计日志也可以统一记录失败原因

### 3.9 审计日志是怎么落库的

这一段非常适合新手理解“自动记录”到底是怎么做到的。

以“后台删除商品”为例：

1. 前端调用 `DELETE /api/admin/products/:id`
2. `adminController.deleteProduct` 先给 `req.auditLog` 写入本次操作的基本信息
   例如模块是 `product`、动作是 `delete`、目标类型是 `product`
3. controller 调用 `adminService.deleteProduct`
4. service 执行真正的删除逻辑
5. 如果删除成功，接口正常返回
6. 响应结束后，`middlewares/auditTrail.js` 会触发
7. `auditTrail.js` 调用 `operationLogService.recordRequestAuditLog(req)`
8. `operationLogService` 把这条记录写进 `operation_logs`

如果第 4 步失败：

- 错误会先进入 `errorHandler`
- `errorHandler` 会把失败原因挂到请求对象上
- 响应结束后 `auditTrail` 仍然会写日志
- 这条日志会被标记为 `failure`

所以你可以把它理解成：

- controller 负责声明“这次操作是什么”
- service 负责执行“这次操作”
- audit middleware 负责在请求结束后统一记账

### 3.10 新手看代码的推荐顺序

如果你刚接手这个项目，推荐按下面顺序看：

1. 先看 `app.js`
   目的：知道项目怎么启动、总路由怎么挂载
2. 再看 `routes/index.js`
   目的：知道后台、商城用户、旧商城接口分别走哪套路由
3. 再看一个具体模块的 route
   例如 `routes/adminRoutes.js`
4. 再看对应 controller
   例如 `controllers/adminController.js`
5. 最后重点看对应 service
   例如 `services/adminService.js`

如果你要排查某个接口，推荐这样找：

1. 先找到接口路径
2. 在 `routes` 里找到对应定义
3. 看它调用了哪个 controller 方法
4. 再看 controller 调用了哪个 service 方法
5. 最后在 service 里看具体业务逻辑和数据库操作

### 3.11 一句话总结整个项目的调用链

可以直接记住这句：

- 请求从 `app.js` 进来，经过 `routes` 分发、`middlewares` 处理、`controllers` 接收、`services` 执行业务、`models/databaseService` 访问数据库，最后再由 `controller + response` 统一返回结果；如果配置了审计信息，响应结束后还会由 `auditTrail + operationLogService` 自动落一条操作日志。
